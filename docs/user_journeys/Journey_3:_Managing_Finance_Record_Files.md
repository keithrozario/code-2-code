**User Journey: Attaching a File to a Transaction**

*   **Start Point:** User is viewing the details of an existing transaction.
*   **End Point:** A file is successfully attached to the transaction and can be viewed or downloaded.

**Flow:**

1.  **User Action:** Clicks the "Attach File" button on the transaction details page.
2.  **System Response:** The system presents a file selection dialog.
3.  **User Action:** Selects a file from their local device and confirms the selection.
4.  **System Response:**
    *   The client-side application initiates an upload of the selected file to the backend.
    *   The backend receives the file and validates it (e.g., file type, size).
    *   If validation is successful, the file is saved, and a record is created linking the file to the transaction.
    *   The transaction details page is updated to show the newly attached file.
5.  **User Action:** Clicks on the attached file's link.
6.  **System Response:** The system serves the file for viewing or download.

**Alternative Paths/Error Handling:**

*   **Invalid File Type:** If the user selects an unsupported file type (e.g., a `.zip` file), the system displays an error message and rejects the upload.
*   **File Too Large:** If the selected file exceeds the maximum allowed size (100MB), the system shows an error and prevents the upload.
*   **Upload Fails:** If the upload process is interrupted, the system displays an error message. No file is attached to the transaction.

**Data Entity: `FlowFile` (Table: `t_flow_file`)**

This entity represents a file attached to a financial transaction.

| Attribute Name | Data Type      | Constraints/Properties                               | Description                                            |
| :------------- | :------------- | :--------------------------------------------------- | :----------------------------------------------------- |
| `id`           | `Integer`      | `PRIMARY KEY`, `Auto-incremented`                    | Unique identifier for the file record.                 |
| `data`         | `byte[]`       | `NOT NULL`, `LONGBLOB`, `FetchType.LAZY`             | The binary content of the file itself.                 |
| `creator`      | `User`         | `NOT NULL`, `FOREIGN KEY` -> `t_user_user(id)`       | The user who uploaded the file.                        |
| `flow`         | `BalanceFlow`  | `FOREIGN KEY` -> `t_user_balance_flow(id)`           | The transaction to which this file is attached.        |
| `createTime`   | `Long`         | `NOT NULL`                                           | The timestamp (in milliseconds) when the file was uploaded. |
| `contentType`  | `String(32)`   | `NOT NULL`                                           | The MIME type of the file (e.g., "image/png").         |
| `size`         | `Long`         | `NOT NULL`                                           | The size of the file in bytes.                         |
| `originalName` | `String(512)`  | `NOT NULL`                                           | The original name of the file on the user's device.    |

**Relationships:**

*   **Many-to-One:** A `FlowFile` belongs to one `User` (the creator).
*   **Many-to-One:** A `FlowFile` belongs to one `BalanceFlow` (the transaction).
*   **One-to-Many:** A `BalanceFlow` can have multiple `FlowFile`s.

**3. Database Tables to be Updated**

*   **Objective:** To identify which database tables are directly impacted by user actions and system processes.
*   **Requirements:**
    *   For each documented user journey, list the specific database tables that are read from or written to.

**User Journey: Managing Finance Record Files**

*   **`t_flow_file`**:
    *   **`INSERT`**: A new record is created in this table when a user successfully uploads a file.
    *   **`SELECT`**: This table is read to retrieve the list of files for a transaction or to fetch the content of a specific file.
    *   **`DELETE`**: A record is deleted from this table when a user removes an attachment.
*   **`t_user_balance_flow`**:
    *   **`SELECT`**: Read to verify the existence and ownership of the transaction before a file can be attached or deleted.
*   **`t_user_user`**:
    *   **`SELECT`**: Read to associate the uploaded file with the current user.
